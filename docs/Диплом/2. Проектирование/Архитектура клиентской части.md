### Сравнение архитектурных стилей
На этапе проектирования клиентской части рассматривались три основных архитектурных подхода: **Clean Architecture (чистая архитектура)**, **модульная архитектура**, **атомарный дизайн (Atomic Design)** и **Feature-Sliced Design (FSD)**. Сравнение проводилось по следующим критериям: специфичность для фронтенда, порог вхождения, скорость добавления новых фич, изоляция кода, тестируемость, стандартизация структуры, масштабируемость


**Таблица 2.х – Сравнение архитектурных подходов**

| Критерий                              | Clean Architecture                                 | Модульная архитектура     | Atomic Design                                | Feature-Sliced Design (FSD)                       |
| ------------------------------------- | -------------------------------------------------- | ------------------------- | -------------------------------------------- | ------------------------------------------------- |
| **Специфичность для фронтенда**       | Низкая (создана для бэкенда)                       | Средняя                   | Высокая (создан для UI)                      | Высокая (разработана для фронтенда)               |
| **Порог вхождения**                   | Высокий (SOLID, DI, абстракции)                    | Средний                   | Низкий (атомы → молекулы → организмы)        | Низкий (фичи как срезы функциональности)          |
| **Скорость добавления новых фич**     | Низкая (требует абстракций)                        | Средняя                   | Средняя (требует разбиения на атомы)         | Высокая (новая фича = новый каталог)              |
| **Изоляция фич**                      | Косвенная (через абстракции)                       | Низкая                    | Низкая (UI-компоненты не изолированы)        | Прямая (правила импортов между слоями)            |
| **Переиспользование между проектами** | Высокая                                            | Средняя                   | Высокая (UI-кит)                             | Средняя                                           |
| **Тестируемость**                     | Высокая (сценарии независимы)                      | Средняя                   | Высокая (компоненты изолированы)             | Высокая (фичи изолированы)                        |
| **Стандартизация структуры**          | Отсутствует                                        | Отсутствует               | Есть (атомы/молекулы/организмы)              | Есть (app/pages/widgets/features/entities/shared) |
| **Связь с предметной областью**       | Слабая                                             | Средняя                   | Слабая (организация по UI, не по бизнесу)    | Сильная (срезы соответствуют реальным функциям)   |
| **Организация кода**                  | По технической роли (сущности, сценарии, адаптеры) | По функциональным модулям | По UI-сложности (атомы, молекулы, организмы) | По бизнес-функциональности (фичи, сущности)       |
| **Масштабируемость**                  | Высокая (слои независимы)                          | Средняя                   | Низкая                                       | Высокая (фичи добавляются без рефакторинга)       |

После сравнения подходов выбран **Feature-Sliced Design (FSD)**. Основные причины:
1. **Связь с предметной областью** – в отличие от атомарного дизайна, который группирует компоненты по UI-сложности (атомы, молекулы), FSD группирует код по бизнес-функциям. Например, всё, что связано с графическим редактором (UI, состояние, API, утилиты), находится в одном каталоге `features/diagram-editor/`. Это упрощает понимание кода новыми разработчиками.
2. **Изоляция фич** – FSD имеет жёсткие правила импортов (слои не могут импортировать из вышележащих), что предотвращает появление циклических зависимостей. Атомарный дизайн таких правил не предоставляет.
3. **Скорость добавления новых нотаций** – каждая новая нотация (BPMN, UML, C4, ER) может быть реализована как отдельная фича без изменения существующего кода. Это соответствует требованиям расширяемости (ФТ-40).
4. **Стандартизация структуры** – FSD предлагает фиксированный набор слоёв (`app`, `pages`, `widgets`, `features`, `entities`, `shared`), что делает проект предсказуемым. Атомарный дизайн определяет только UI-слой, оставляя остальную архитектуру на усмотрение разработчика.
5. **Чёткое разделение UI и бизнес-логики** – в FSD UI-компоненты (кнопки, поля ввода) находятся в `shared/ui`, а бизнес-сущности – в `entities`, а пользовательские сценарии – в `features`. Атомарный дизайн смешивает UI и логику, так как не определяет, где должны храниться API-вызовы или состояние.
**Ограничения Clean Architecture:**
- Clean Architecture требует создания множества абстракций (сущности → сценарии → адаптеры → фреймворки). В React-приложении большая часть логики связана с отображлением UI, эвентами и состоянием, которые сложно отделить абстракциями без избыточного кода.
- Отсутствие единых соглашений приводит к тому, что разные разработчики могут реализовывать одну и ту же функциональность по-разному.
- Порог вхождения значительно выше, что критично для учебного проекта.
**Ограничения модульной архитектуры:**
- Модульная архитектура не даёт чётких правил изоляции между модулями, что приводит к «спагетти-коду» при масштабировании.
- Отсутствие стандартизации структуры – каждый модуль может быть организован по-своему.
**Ограничения атомарного дизайна в чистом виде:**
- Не решает проблему организации бизнес-логики – только описывает UI-компоненты.
- Не обеспечивает изоляцию фич – компоненты разных фич могут перемешиваться.    
- При масштабировании каталог `atoms/` становится неуправляемым.


### Правила организации импортов

Для обеспечения изоляции слоёв установлены следующие правила импортов:
1. **Запрещены относительные импорты между слоями** – все импорты должны проходить через публичные API (`index.ts`) каждого слоя.
2. **Слой не может импортировать из вышележащих слоёв** – например, `features` не может импортировать из `pages`.
3. **`shared/ui` не зависит от бизнес-логики** – компоненты UI-кита не должны знать о сущностях или фичах.
4. **`entities` может импортировать только `shared`** – сущности не зависят от конкретных фич.
5. **`features` может импортировать `entities` и `shared`** – фичи оперируют сущностями и используют UI-кит.
