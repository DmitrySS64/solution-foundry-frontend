# Графический редактор нотаций на базе Konva

## Архитектурные принципы и слоистая структура

Редактор построен согласно принципу **разделения ответственности** на независимые визуальные слои. Каждый слой отвечает за конкретный набор объектов и имеет собственный `Konva.Layer`, что позволяет минимизировать количество перерисовок: обновления ограничиваются только тем слоем, в котором произошло изменение.

- **GridLayer** – фон с тонкой сеткой, отрисовывается один раз и использует `batchDraw` для быстрых обновлений при масштабировании.
- **EdgeLayer** – содержит все `EdgeRenderer`‑элементы (`Konva.Line`, `Konva.Arrow`). Рёбра подключаются к `edgeRegistry`, что позволяет напрямую менять их координаты без обращения к React‑дереву.
- **NodeLayer** – хранит `ShapeRenderer`‑компоненты, представляющие узлы как `Konva.Group` с вложенными примитивами (rect, circle, diamond, text, svg). Каждый узел регистрируется в `nodeRegistry`.
- **TransformerLayer** – отдельный `Konva.Transformer`, отображающий рамку изменения размеров для выбранного узла. Включает ограничения `canStretch` и `preserveAspectRatio` из модели узла.
- **Overlay‑слои** (highlight, anchors, selection) – визуальные помощники, которые находятся выше всех остальных и не попадают в экспортируемый документ.

Такое слоистое построение гарантирует, что интерактивные элементы (например, `Transformer`) всегда находятся над базовым контентом, а изменение одного слоя (например, перемещение узла) не приводит к повторному рендерингу сетки или остальных узлов.

## Механизмы взаимодействия и рендеринга

**React Konva** обеспечивает декларативный мост между React‑компонентами и Konva‑объектами. Основной цикл выглядит так:

1. Пользователь инициирует действие (drag, click, resize) → генерируется событие Konva.
2. Обработчики в `canvas`‑компонентах (например, `DiagramCanvas.onNodeDragMove`) вызывают утилиту `syncEdgeKonva`, которая **мутирует** только `Konva`‑объекты ребёр, используя их ссылки из `edgeRegistry`. Это происходит без запуска нового рендера React, что сохраняет высокую частоту кадров.
3. При окончании drag‑операции (`onNodeDragEnd`) вызывается действие `updateNode` из Zustand‑хранилища, которое **иммутабельно** обновляет `DiagramNode` в `document`‑слайсе и фиксирует снимок в истории (`pushPastSnapshot`).
4. Изменения в Zustand автоматически пробрасываются в компоненты через `useEditorStore.selectors`, ограничивая перерисовку только теми компонентами, которые действительно читают изменённые части стейта (например, `ShapeRenderer` читает `node`‑данные, `EdgeRenderer` – `edge`‑данные).

Все пользовательские операции (добавление/удаление узлов и рёбер, изменение свойств) инкапсулированы в **централизованные actions** (`addNode`, `deleteNode`, `addEdge`, `updateEdge`, `setViewport`, `undo`, `redo`). Это упрощает тестирование и делает логику предсказуемой. Для повышения отзывчивости используются `requestAnimationFrame`‑балансировщики (`batchDraw`) внутри Konva, а также `useDeferredValue`/`useTransition`‑парадигмы там, где требуется отложить тяжёлые вычисления (например, пересчёт геометрии рёбер).

## Декларативная система нотаций

Нотации хранятся в `public/notations/*.json` и имеют строгую схему:

```json
{
  "id": "bpmn",
  "name": "BPMN 2.0",
  "version": "1.0",
  "elements": [
    {
      "type": "bpmn-task",
      "name": "Задача",
      "extends": "shape",
      "icon": "mdiSquare",
      "defaults": {
        "shapeType": "rectangle",
        "width": 140,
        "height": 70,
        "style": { "fill": "#DBEAFE", "stroke": "#3B82F6", "strokeWidth": 2, "opacity": 1 }
      },
      "properties": [
        { "name": "label", "label": "Название", "type": "text", "default": "Новая задача" },
        { "name": "taskType", "label": "Тип задачи", "type": "select", "options": ["user","service"], "default": "user" }
      ]
    }
  ],
  "connections": {
    "allowedTypes": ["bpmn-task","bpmn-gateway","bpmn-event"],
    "edgeDefaults": { "stroke": "#475569", "strokeWidth": 2, "endArrow": "default" }
  },
  "validationRules": [
    { "rule": "startEventRequired", "message": "Диаграмма должна содержать начальное событие" }
  ]
}
```

**Ключевые разделы**:
- `elements` – массив описаний фигур; каждый элемент содержит `type`, `name`, `extends` (определяет базовый набор свойств), `icon`, `defaults` (начальные размеры и стили) и `properties` (массив пользовательских параметров, используемых в инспекторе).
- `connections` – правила, какие типы узлов могут быть соединены, и набор дефолтных стилей для рёбер.
- `validationRules` – список проверок, которые выполняются при сохранении диаграммы.

Во время **инициализации** `notationRegistry.ts` происходит:
1. **Динамический импорт** всех JSON‑файлов через `import.meta.globEager`.
2. **Нормализация**: каждому элементу присваивается уникальный `notation.id` вида `${notationId}.${element.type}`.
3. **Создание primitives**: если в `defaults` указано `shapeType`, генерируется базовый примитив (rect, circle, diamond). При отсутствии `primitives` в JSON используется fallback‑primitive из `defaults.shapeType`.
4. **Формирование `box`** – начальные размеры и ограничения `canStretch`/`preserveAspectRatio`.
5. **Экспорт** `NotationDefinition[]`, которые затем объединяются с базовыми фигурами (`rectangle`, `circle`, `diamond`) в `nodeDefinitions.ts`. Благодаря этому редактор автоматически отображает новые типы в `DiagramSidebar` без изменения кода.

## Состояние и синхронизация

Zustand‑хранилище разбито на несколько **слайсов**:
- **documentSlice** – массивы `nodes` и `edges`, а также метаданные (`createdAt`, `updatedAt`).
- **historySlice** – стек `past` и `future` для реализации `undo/redo`.
- **selectionSlice** – текущий набор выбранных `id` (узлы/рёбра).
- **viewportSlice** – параметры `x`, `y`, `zoom` и их методы `setViewport`.
- **interactionSlice** – флаги, определяющие состояние перетаскивания, соединения, подсветки анкоров.

Все изменения происходят через **action‑функции** (`addNode`, `updateNode`, `deleteNode`, `addEdge`, `updateEdge`, `setViewport`, `undo`, `redo`). Каждая action, изменяющая документ, сохраняет **снимок** в `historySlice` через `pushPastSnapshot`, после чего сбрасывает `future`.

Для **коллаборативного редактирования** используется **Yjs**: состояние `documentSlice` синхронизируется с `Y.Map` в `yjsProvider.ts`. При каждом изменении Zustand генерирует `Yjs`‑транзакцию, а изменения, полученные из Yjs, пробрасываются обратно в Zustand через `applyYjsUpdate`. Конфликты решаются встроенным CRDT‑механизмом Yjs, обеспечивая согласованность модели у всех участников.

**Сериализация** реализована в `model/serializers/document.serializer.ts`. Функция `serialize(document)` преобразует текущий `DiagramDocument` в чистый JSON, включая свойства узлов, рёбра, их стили и пользовательские свойства нотаций. `deserialize(raw)` восстанавливает структуру, но также **нормализует**: восстанавливает связи между узлами и ребрами, создает недостающие `edges`‑индексы и заполняет отсутствующие поля значениями по умолчанию.

Эти механизмы совместно обеспечивают **производительность** (минимальные перерисовки, batchDraw, регистры), **расширяемость** (добавление новых нотаций через JSON) и **надёжность** (история, коллаборация, валидация) графического редактора.

## Старое

Графический редактор нотаций реализован с использованием Konva и React Konva. Эти библиотеки позволяют строить интерактивный 2D-холст, сохранять декларативный стиль описания элементов, привычный для React‑приложений. Редактор предназначен для создания диаграмм, содержащих узлы, связи, подписи и элементы выбранной нотации.

Холст разделен на несколько логических слоев. Отдельно обрабатываются сетка, связи, узлы, элементы трансформации и оверлейные элементы. Такое разделение упрощает управление порядком отрисовки: связи располагаются под узлами, трансформер отображается поверх выбранного объекта, а вспомогательные элементы не смешиваются с содержимым документа.

Основные операции редактора включают добавление узлов, перемещение элементов, изменение размеров, выделение, удаление и создание связей. Для связей поддерживаются начальная и конечная точки, привязка к узлам, управляющие точки и визуальные маркеры концов. При перемещении узла связанные линии перерасчитываются, что обеспечивает согласованность диаграммы.

Расширяемость редактора достигается через декларативную систему нотаций. Описания BPMN, UML и C4 загружаются из JSON‑файлов, после чего преобразуются в определения элементов. В описании элемента задаются тип, название, начальные размеры, свойства и набор графических примитивов. Благодаря этому новые элементы могут добавляться без переписывания базовой логики холста.

Инспектор свойств отображает редактируемые параметры выбранного элемента: текстовую метку, цвета, числовые значения, варианты из списка и другие настройки, предусмотренные конкретной нотацией. Изменения применяются к модели узла или связи и сразу отражаются на холсте.

Состояние диаграммы хранится в Zustand‑хранилище и синхронно с совместным Yjs‑документом. Документ представлен массивами узлов и связей, а дополнительные части состояния отвечают за выделение, масштаб, смещение холста и историю изменений. Для сохранения используется сериализация документа в JSON, что делает модель пригодной для локального хранения и передачи на сервер.
